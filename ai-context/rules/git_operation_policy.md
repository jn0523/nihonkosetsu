# Git運用ポリシー（恒久ルール）

目的: Git運用ルールを実装計画書ではなく恒久ルールとして固定し、全エディタが常時参照できる状態にする。

## 1. 基本方針
- Git運用ルールは本ファイルを正本とし、個別の実装計画書に重複定義しない。
- 実装計画書では「本ポリシー参照」を記載し、差分が必要な場合のみ例外として追記する。

## 2. ブランチ運用
- `main` への直接コミットは禁止。
- 作業は `feature/*` `bugfix/*` `hotfix/*` `release/*` のいずれかで行う。
- リリース対象は `release/*` ブランチで確定し、`main` への反映前にリリース前検査を必須とする。

## 3. マージ・プッシュ前の必須確認
- 変更概要、影響範囲、ロールバック手順を明記する。
- テスト結果（最低: 主要シナリオの再現確認）を残す。
- シークレットや個人情報の混入がないことを確認する。
- AIガバナンス監査結果（PASS/CONDITIONAL/FAIL）を確認する。

## 4. release -> main 反映ルール
- `release/*` から `main` へ push / merge する前に、`ai-context/workflows/release_gate.md` を実行する。
- `release_gate` で `FAIL` が出た場合は反映不可。
- `CONDITIONAL` の場合は、条件・期限・担当者を確定してから反映可否を再判定する。

## 5. コミットメッセージ
- Conventional Commitsを推奨。
- 曖昧語（`update` `fix` `変更` `対応`）のみの説明は禁止。

## 6. 監査証跡
- 主要なマージ判断（PASS/CONDITIONAL/FAIL、例外承認、ロールバック判断）は記録を残す。
- 監査証跡は運用ドキュメントまたはリリース記録に紐づける。
