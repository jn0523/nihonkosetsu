# Goal

図を継続的に改善できる運用にし、設計・運用ドキュメントの更新コストを下げる。

# When to Use

- 新規のアーキテクチャ図を作る。
- 運用フロー図を差分更新する。
- 既存図をレビューして読みやすさを改善する。

# When NOT to Use

- 一時的なメモ図で保存不要な時。
- 機密情報を含む図を外部共有する必要がある時。

# Trigger Templates

以下の依頼文をそのまま使うと、作図と更新の意図が明確になる。

- 「アーキテクチャ図を新規作成して。反復で整えて保存して」
- 「この運用フロー図を差分更新して、JSONを再保存して」
- 「図の視認性レビューをして、線交差とラベル過密を改善して」
- 「1図1目的で再構成して、役割ごとに視覚分離して」
- 「図の命名規約は docs/diagrams/NAMING_TEMPLATE.md に合わせて」

# Security Guardrails

1. 認証なしAPI前提の運用は localhost 限定にする。
2. 図にトークン、鍵、個人情報を含めない。
3. 外部公開が必要な場合は別途アクセス制御を設計する。

# Iteration Loop

1. Plan:
- 図の目的を1文で定義する（境界可視化、障害対応、責任分界など）。

2. Draw:
- 要素を最小単位で配置し、関係線を明確化する。

3. Inspect:
- 図全体の整合を確認する。
- 要素の重なり、視線の流れ、ラベルの過密を確認する。

4. Refine:
- 整列、分配、グルーピングで視認性を上げる。
- 役割ごとに色と形を統一して解釈コストを下げる。

5. Persist:
- JSONを保存して再利用可能にする。
- 差分更新時は既存JSONを読み込み、変更箇所のみ修正する。

# Diagram Quality Rules

- 1図1目的を原則にする。
- テキスト量より関係性の明確さを優先する。
- 交差線を減らし、左から右または上から下に流れを統一する。
- 役割（ユーザー/アプリ/API/データ基盤/外部SaaS）を視覚的に分離する。

# 出力形式

## 図解運用結果
- 目的
- 実施した変更
- 保存した成果物
- 次回更新トリガー
